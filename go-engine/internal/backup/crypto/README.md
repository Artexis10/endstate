All `Decrypt*` and `Unwrap*` paths return `ErrAEADAuthFailed` on AEAD authentication failure. The two failure modes "wrong key" and "tampered ciphertext" are intentionally indistinguishable to callers â€” both indicate "do not use this output."

## Locked invariants

- **KDF parameters** are pinned to the v1 floor (Argon2id, 64 MiB, 3 iterations, parallelism 4, 64-byte output, 16-byte salt). Sub-floor parameters are refused with `ErrKDFParamsBelowFloor` regardless of what a server advertises.
- **Nonces** are always 12 bytes from `crypto/rand`. Never derived. Never reused.
- **AAD** binds the chunk index (4-byte big-endian uint32) inside chunk ciphertexts, and binds the sentinel `0xFFFFFFFF` inside manifest ciphertexts. A manifest blob will not decrypt as chunk index 0 and vice versa.
- **Wire format** is `nonce || ciphertext || tag` for every AEAD output (RFC 5116 layout).
- **Wrapped DEK** is exactly 60 bytes (12 nonce + 32 ciphertext + 16 tag).
- **Recovery mnemonic** is always 24 words (256 bits of entropy + 8-bit BIP39 checksum). Bad-checksum phrases are rejected.

## The two `0xFFFF`-shaped sentinels in the contract

`0xFFFFFFFF` (this package's `ManifestAAD`) and `chunkIndex == -1` (the wire flag in API presigned-URL responses, contract Â§7) are independent. One is cryptographic binding inside the encrypted blob; the other is a transport-layer marker for "this URL targets the manifest." They are unrelated and code MUST NOT conflate them.

## Library boundaries

Only:

- Go stdlib (`crypto/aes`, `crypto/cipher`, `crypto/rand`, `encoding/binary`)
- `golang.org/x/crypto/argon2`
- `github.com/tyler-smith/go-bip39`

No third-party "crypto utility" libraries. Each additional dependency is a supply-chain risk.

## Memory hygiene

`zero.go` provides `zeroBytes(b []byte)` and is applied to derived intermediate buffers (the 64-byte Argon2id output before it is split into `ServerPassword`/`MasterKey`, and the recovery KDF output). Go's escape analysis, garbage collection, and stack/heap behaviour limit how strong this guarantee is â€” a sensitive value may have been copied elsewhere before zeroing runs. The explicit zeroization is defense-in-depth and a code-review signal, not a hard memory-cleansing primitive. The doc comment on `zeroBytes` says so explicitly.

## Test vectors

`testdata/vectors.json` contains byte-precise vectors generated by an independent Python reference (`scripts/generate-crypto-vectors.py`, using `pyca/cryptography`, `argon2-cffi`, and `python-mnemonic`). The Go test suite asserts byte-equality against the JSON.

The vector generator is deterministic â€” re-running it produces the same JSON. The vectors test the *decrypt* path against fixed Python-sealed ciphertexts. Encrypt-side coverage is provided by Go-only roundtrip tests (since the engine generates a fresh nonce per call, and asserting equality against a fixed ciphertext would require seeding the engine's CSPRNG, which we deliberately do not do).

To regenerate:

```bash
python -m venv .venv && .venv/Scripts/pip install -r scripts/requirements.txt
.venv/Scripts/python scripts/generate-crypto-vectors.py
```

Then `go test ./internal/backup/crypto/...` should still pass without `-short`.

## Tier of tests

- **Default test suite** (no `-short`): runs the full v1-floor KDF and the Python-vector cross-check. ~0.5â€“1 second on a modern CPU.
- **`-short`** (fast iteration loops): skips the v1-floor KDF tests and the Python-vector cross-check. Roundtrip and AAD-binding tests still run.

CI runs the default suite (no `-short`), so the v1-floor KDF and the Python cross-check are exercised on every push.
