## Why

`add-backup-auth-client` (PR #19) and `add-backup-storage-client` (PR #22) shipped the Hosted Backup auth flow, the storage client, and every command handler. Both intentionally left `go-engine/internal/backup/crypto/` as stubs that return `ErrNotImplemented`; consumers short-circuit on that sentinel and surface a "crypto module not yet implemented" error.

This change replaces the stub bodies with real cryptography implementing the locked sections of `docs/contracts/hosted-backup-contract.md` (┬¦1, ┬¦2, ┬¦3, ┬¦6). It is delivered as an isolated PR so the cryptographic surface gets focused review against the contract and the relevant standards (RFC 9106, RFC 5116, NIST SP 800-38D, OWASP Cryptographic Storage Cheat Sheet).

The integration surface is unchanged. Every function signature already in `crypto/types.go`, `kdf.go`, `dek.go`, `envelope.go`, `recovery.go` is preserved verbatim ŌĆö only the bodies change. No call site moves.

The two contract sentinels that look superficially similar (the AAD `0xFFFFFFFF` from ┬¦3 inside the encrypted manifest blob; the wire flag `chunkIndex == -1` from ┬¦7 in API URL responses) are independent and stay independent in this code.

## What Changes

- Replace stub bodies in `internal/backup/crypto/` with real implementations:
  - **KDF**: Argon2id via `golang.org/x/crypto/argon2.IDKey`, 64-byte output split into `serverPassword[0:32]` + `masterKey[32:64]`, parameter floor enforced client-side
  - **DEK**: 32-byte CSPRNG generation; AES-256-GCM wrap/unwrap with `masterKey`
  - **Chunks**: AES-256-GCM with fresh 12-byte CSPRNG nonce per chunk and 4-byte big-endian `chunkIndex` as AAD
  - **Manifest**: same primitive as chunks but with `0xFFFFFFFF` sentinel AAD
  - **Recovery key**: 32 bytes CSPRNG ŌåÆ 24-word BIP39 (`tyler-smith/go-bip39`) ŌåÆ KDF-derived recovery key + server-side verifier
- New `zero.go` with a small zeroization helper used after handling sensitive material
- Test suite covering: roundtrips, parameter-floor rejection, salt-length rejection, nonce uniqueness, AAD binding, ciphertext/tag tamper rejection, manifest-vs-chunk-0 distinguishability, BIP39 checksum validation, and Python-cross-checked test vectors
- Test vectors (`testdata/vectors.json`) deterministically generated by `scripts/generate-crypto-vectors.py` (Python `pyca/cryptography` + `argon2-cffi`); Go tests assert byte-equality against the JSON fixture
- Two new direct dependencies: `golang.org/x/crypto` and `github.com/tyler-smith/go-bip39`

## Capabilities

### New Capabilities

- `hosted-backup-crypto-module`: Client-side cryptographic primitives for Endstate Hosted Backup ŌĆö Argon2id KDF, AES-256-GCM chunk and manifest encryption, AES-256-GCM DEK wrapping, BIP39 recovery key generation/parsing, recovery KDF, and server-side recovery-key verifier production.

### Modified Capabilities

None. The existing capabilities `hosted-backup-auth-client` and `hosted-backup-storage-client` consume this module unchanged.
