## Context

- The home-manager apply path is generated-and-inspectable: `GenerateHomeFlakeFromSettings` →
  `CompileHomeNix` (`internal/realizer/nix/home_catalog.go`) renders a `home.nix` from the declared
  catalog and `writeHomeFlake` (`home_flake.go`) wraps it in a commented "GENERATED BY ENDSTATE"
  `flake.nix`, copying the config in (`./home.nix`) and staging declared `files/` beside it. The
  directory **persists, is regenerated each run, and is explicitly ejectable** (the in-flake comment
  invites `nix run home-manager -- switch --flake .#<name>`). Treat it as world-readable and
  potentially committed to a dotfiles repo.
- The declarative input is `manifest.HomeManagerSettings` (`internal/manifest/types.go`): curated
  `git` / `shell` / `direnv` / `starship`, a raw `programs` passthrough, and a `files` map
  (target → source). A future `homeManager.secrets` input attaches to `HomeManagerConfig` next to
  `Flake` / `Config` / `Settings`.
- `capture` re-emits `homeManager.settings` from provisioning history
  (`nix-home-manager-catalog-capture`). Any secret that reached `settings` would round-trip as
  plaintext — so capture redaction is part of *this* design, not a later afterthought.
- Endstate already has a "no auto-secret capture" posture elsewhere: `module.secrets.files` are
  excluded from capture zips (`capture-bundle-zip`), and restore warns on sensitive target paths
  (`.ssh`, `.aws`, `.gnupg`, credentials, tokens). The home-manager secrets story should be
  consistent with that posture: **the engine handles references, never material.**

## Goals / Non-Goals

**Goals (of the eventual feature this scopes):**

- A user can have secret-dependent home-manager config without the secret plaintext ever entering
  the generated `flake.nix` / `home.nix` / staged files.
- `capture` of a secrets-bearing machine emits the secret *reference*, never the material.
- Cross-OS: the model works on both Linux and macOS (the realizer's two targets).
- Minimum new dependency surface; the most conservative option ships first.

**Non-Goals:**

- No implementation in this change. No Go, no engine wiring, no real backend integration.
- Not selecting the final backend here — the recommendation is a starting position; the decision is
  the human's (see Open Questions + tasks.md).
- Not a general secret-management product (no Endstate-hosted KMS, no key custody). Endstate
  *references* secrets a user already manages; it is not a vault.
- Not Windows. The home-manager path is realizer-only (Linux/macOS); winget ignores `homeManager`.

## The invariant that frames everything

**Referenced, never embedded.** The generated artifacts are inspectable and committable; therefore
the *only* thing about a secret that may appear in them is a **reference** — a path the secret is
decrypted to at activation, an environment variable name it is exposed under, or an external store
handle. The plaintext (and any decryptable ciphertext the engine itself produced) must never be
written by the generator. This is the line every option below is measured against.

## Three-way comparison

| Dimension | **sops-nix** | **agenix** | **Documented boundary** (no new dep) |
|---|---|---|---|
| Model | age/PGP-encrypted secret files committed to the repo; a home-manager module decrypts them at activation to a runtime path | age-encrypted secret files committed to the repo; thin home-manager module decrypts to a runtime path | Engine never touches secret material; user provisions the secret out-of-band; generated config only references the expected path / env var |
| Where ciphertext lives | `secrets/*.yaml` (or `.json`/`.env`/binary) committed beside the flake | `secrets/*.age` committed beside the flake | Nowhere in Endstate's tree — entirely user-owned |
| What the **engine generates** | flake input `sops-nix`, a `sops.secrets.<n>` block, and reference-only `home.*` wiring (`config.sops.secrets.<n>.path`) | flake input `agenix`, an `age.secrets.<n>` block, reference-only `home.*` wiring (`config.age.secrets.<n>.path`) | only the reference: `home.file`/`sessionVariables` pointing at a user-declared path / env var; possibly a doc-comment noting the contract |
| What stays **user-owned** | the age/PGP **private key** (host key or `~/.config/sops/age/keys.txt`) — never generated, never captured; plus authoring the encrypted files | the age **identity** (`~/.ssh` key or age key) — never generated, never captured; plus authoring `.age` files | **everything secret-related**: encryption, key custody, delivery, the file at the referenced path |
| Inspectability / auditability | High: generated flake is still fully readable; committed secrets are ciphertext (safe). Decryption keys are out-of-tree. The reference path is visible and auditable | High: same — readable flake, ciphertext at rest, out-of-tree key. Smaller module = less to audit | Highest: generated flake contains only a path/env reference; zero ciphertext, zero crypto wiring to reason about. Nothing to leak because the engine holds nothing |
| Dependency cost | New flake input `sops-nix` + transitive `sops`/`age`; a home-manager module; key-provisioning UX the engine must document (and ideally not own) | New flake input `agenix` + `age`; a smaller module; same key-provisioning UX | **Zero** new flake inputs, zero crypto deps. Pure documentation + a typed reference field |
| Cross-OS (Linux/macOS) | Works on both; macOS needs an age key (no system host key by default), so key bootstrap differs by OS — a doc/UX wrinkle | Works on both; same macOS key-bootstrap wrinkle; agenix's reliance on SSH host keys is more Linux-centric, so the age-identity path is the cross-OS-safe one | Trivially cross-OS: the engine asserts a path; the user satisfies it however their OS allows |
| Capture / round-trip | Capture emits the secret **reference + the path to the committed ciphertext file**, never plaintext, never the private key. The `.sops.yaml` recipients are non-secret and may be referenced | Same: capture emits the reference + the `.age` file path; never plaintext, never the identity | Capture emits only the reference (path/env). There is no Endstate-held material to accidentally emit — the safest capture story |
| Failure mode if engine misbehaves | A generator bug could in principle read a decrypted value into output → must be structurally prevented (generator never opens the plaintext) | Same risk, smaller code | **No such risk**: the engine never has the material to leak |
| Maturity / ecosystem | Most widely used NixOS/home-manager secrets solution; large community; richer features (formats, templates, owner/mode) | Popular, minimal, well-regarded; fewer features | n/a (it's a contract, not a tool) |

## Recommendation

**Phase 1: ship the documented boundary as the default and only option. Phase 2: add a typed
backend (lean agenix) once the boundary is proven and a typed `homeManager.secrets` input is
designed.**

Rationale, tied to invisible-but-inspectable:

1. **The boundary is the strongest possible expression of "referenced, never embedded."** If the
   engine never holds secret material, it *cannot* leak it into the generated flake or a captured
   manifest. Every other option must *structurally prove* the generator never reads plaintext into
   output; the boundary makes that proof trivial because there is nothing to read. For a tool whose
   core promise is "the generated config is safe to inspect and commit," starting where the engine
   holds zero secret material is the conservative, principle-aligned default.
2. **Zero new dependency cost.** sops-nix/agenix each add a flake input, a crypto toolchain, a
   home-manager module, and OS-specific key-bootstrap UX. The boundary adds a typed reference field
   and documentation. Endstate's posture elsewhere (no auto-secret capture; warn-don't-own on
   sensitive paths) is already "the engine does not own secrets" — the boundary is continuous with
   it.
3. **It de-risks the capture round-trip immediately.** The headline danger is `capture` re-emitting
   a secret that reached `settings`. With the boundary there is no Endstate-held material to emit;
   capture carries only the reference. That invariant can be specified and enforced now, before any
   crypto backend exists.
4. **When a managed backend is wanted, lean toward agenix over sops-nix for Phase 2.** agenix's
   smaller surface is easier to audit (fewer lines between the user and the ciphertext), and the
   age-**identity** path (rather than agenix's SSH-host-key default) is the cross-OS-safe choice for
   the Linux+macOS realizer. sops-nix's richer feature set (templates, formats, owner/mode) is real,
   but those features are not needed for the first secrets a home config wants (a key file, an env
   token), and the extra surface cuts against auditability. Revisit sops-nix only if a concrete need
   (templating, multi-format) appears.

In short: **boundary now, agenix later, sops-nix only on demonstrated need** — least dependency and
least leak-surface first, richer managed backends only when justified.

## Phased path

- **Phase 1 (documented boundary).** Introduce a typed reference-only input (sketch:
  `homeManager.secrets[].{name, path | env, source?: "external"}`) that the generator turns into
  reference-only `home.*` wiring and that capture carries verbatim. The engine asserts the reference;
  the user satisfies the path/env out-of-band. Spec + enforce the **no-embed** and
  **capture-no-material** invariants. Document the contract (engine references, user provisions).
- **Phase 2 (managed backend, lean agenix).** Add `backend: "agenix"` to the typed input: the engine
  generates the flake input + `age.secrets.<n>` block + reference wiring, pointing
  `home.*` at `config.age.secrets.<n>.path`. Ciphertext files are user-authored and committed; the
  age identity is user-owned, never generated, never captured. Capture emits the reference + the
  ciphertext file path. The no-embed invariant is enforced structurally: the generator never opens a
  decrypted value.
- **Phase 3 (optional, sops-nix).** Add `backend: "sops-nix"` only if a concrete need (templating,
  multi-format, owner/mode) is demonstrated. Same invariants, larger module.

## Open Questions (for the human)

1. **Default backend.** Confirm boundary-first. Or is there appetite to ship a managed backend in
   Phase 1 directly (and if so, agenix or sops-nix)?
2. **Typed input shape.** Is `homeManager.secrets` a list of `{name, path|env, backend?, source?}`
   the right schema? Should it live under `settings` or as a sibling of `Flake`/`Config`/`Settings`?
3. **Reference surface.** Which `home.*` targets may a secret reference flow into — only `home.file`
   source + `home.sessionVariables` values, or broader (e.g. inside the raw `programs` passthrough)?
   The broader the surface, the harder the no-embed invariant is to enforce mechanically.
4. **Capture of the ciphertext file (managed backends).** Should capture record the committed
   `.age`/`.sops` file *path* only, or also copy the ciphertext into the captured bundle? (Ciphertext
   is safe at rest, but copying expands what the engine touches.)
5. **Key bootstrap UX.** For a managed backend, the user-owned key (age identity) is never engine-
   generated — but should the engine *detect and warn* when the referenced identity is absent at
   activation (fail-fast), or stay silent (pure boundary)?
6. **macOS specifics.** Confirm the age-identity path (not SSH host keys) as the cross-OS default for
   any managed backend, given macOS has no default system host key.

## Risks / Verification (of the eventual feature)

- **Leak via the generator** — the single highest risk for managed backends: a code path that reads a
  decrypted secret into generated output. Mitigation: the generator must be structurally incapable of
  reading plaintext (it wires *references* only); a test asserts no secret material appears in
  `flake.nix` / `home.nix` / staged files for any input.
- **Leak via capture** — a secret reaching the captured manifest. Mitigation: capture carries only
  the reference; a test asserts a captured manifest from a secrets-bearing machine contains no
  plaintext.
- **Inspectability regression** — a backend that makes the generated flake unreadable or non-
  ejectable. Mitigation: the generated flake stays commented and self-contained; ciphertext lives in
  separate committed files, not inlined.
- These are verification *targets for the future implementation*, recorded so the eventual proposal
  inherits them. **No code is verified by this design-only change.**
